Unity Implementation

Each scene can be separated into the following parts: The camera, base scene components, optional scene components, the user interface and its containing scripts.

Camera Scripts

The camera contains two scripts, the first for camera movement. There are two methods to enable and disable camera movement, which are used to disable movement while the user is in certain menus e.g. help menu. Camera movement and rotation sensitivity controls change the multiplier by which the camera is moved or rotated every frame. The camera movement script uses Unity’s rigidbody physics and mesh colliders on objects in the scene to prevent the camera escaping. The second script is for saving and restoring camera positions within the scene. It uses number keys 1 through 0, holding for 2 seconds to save a position and simply pressing it to restore. This was done to allow the user to assign hotkeys within OBS on the same key to simultaneously change OBS scenes, where positioning of presenters is arranged to work with that specific camera angle.

Scene Components

Base components are the assets and models always in the scene that cannot be customized by the user.
Optional scene components are ones that can be customised by the user. Our implementation allows us to easily add such components, it is designed for fast development in any future work. Each scene has an empty GameObject named “Optional”, in which child GameObjects of optional components are placed. The child GameObjects have a ComponentInfo script, which defines their properties - the friendly name (displayed to users in UIs), an internal name (used for saving scene configurations), and a boolean defining if the component should be enabled by default.

Optional Component Configuration

The optional components configuration UI, driven by SetupScript, iterates over the child GameObjects in Optional, and creates toggles using information from ComponentInfo to allow the user to turn it on or off. A dictionary is used to store relations between toggle objects and the corresponding optional GameObjects they control. A listener is added to the toggle that does a lookup in this dictionary and toggles the active state of the retrieved GameObject.
Optional components may need to be configurable by the user, such as selecting which desktop to capture for the desktop capture component, so the IComponentConfigurator interface defines such behaviour, allowing future developers to easily add configurable components. The configurator will draw an additional sub-UI in the scene component configuration UI, and internally update the scene based on user interaction with the sub-UI. The interface defines two methods, one to get the size of the sub-UI needed for correct spacing in the parent UI, and the second to trigger the generation of the sub-UI at the given position. Implementing scripts, such as DesktopConfigurator, are placed in the optional GameObject, and are automatically detected by the UI generator.

Desktop Capture

Desktop capture is achieved through the uDesktopDuplication library, which allows us to interface with DirectX Graphics API to display the user’s desktop as a texture in the Unity scene. DesktopConfigurator class implements IComponentConfigurator, it allows the user to select which desktop will be captured and displayed in the environment.

User Interface

The UIs in each presentation scene consist of 4 prefabs, scene configuration menu, drop-down menu, help menu, and load menu. This was done to maximize reusability and reduce needed configuration throughout the scenes. In general, each prefab has a script which executes functions triggered on UI events such as pressing a button. UI prefabs added to scenes require some configuration such as supplying the Camera object and links to other UIs.

Save and load system

The save and load system works by saving to a json format, containing the environment name, a list of enabled optional components, and the saved camera angles. Save files are stored in the Unity persistent data folder. SaveObject is an intermediate data object which is serialized into the json using Unity’s JsonUtility. SaveUtil class handles saving, loading and getting a list of available saves to load. The LoadSettings object is used to pass information between scenes on whether a save should be loaded upon scene initialization. When a save is selected in the load menu, the name of the selected save file is saved in this object, and the scene is loaded. During the SetupScript of the loaded scene, the save file is detected and loaded: camera angle bank is loaded and optional components are enabled where defined. The user still has a chance to edit the configuration as they would in a normal scene setup.

Multiple presenters

Our multiple presenters feature is provided by our WebRTC based Express.js web app. The app works with different video call “rooms” which corresponds to the URL

Server side:

On connection to the website, a uuid v4 is generated and this becomes the “roomid”. The url also redirects to this uuid. Socket.IO is then used to monitor all new connections
Each user receives a peerid from the PeerJS server on connection to a call. This peerid is then distributed to all peers in the room using Socket.IO.
The relay server also distributes the BodyPix ml model to all peers.

Client side:

When the client joins a call a new peerjs peer object is created and assigned to them. This includes a peerid which is then shared with the relay server and broadcasted to all other peers in that room.
On receipt of a peerid from the relay server, a call is initiated with that peer which includes a video stream exchange between each peer in the call.

Background removal

Per 120 milliseconds, a video frame is processed using the BodyPix model to produce a mask. This mask is then applied to the user’s video stream and the space outside the mask but within the canvas is filled with green. The result is an algorithmic background removal.
