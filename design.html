UI Design



Design Principles


Visibility:
We tried very hard in our design to make things like buttons very obviously clickable (by highlighting them when hovered over), scrollable panels very obviously scrollable (by including a scrollbar to the side), and so on with anything else interactable in the UI.

Feedback:
There are visibly different button states (a button changes colour when it’s pressed) and different toggle states (a toggle either changes colour or a tick box is ticked) so that it is clear what is selected and what isn’t to the user.

Affordance:
We tried to make the virtual affordance of our program good by having clear visual indicators of what to do, for example back and forward arrows. We also included a clear set of instructions in the program accessible at any time in a scene to improve the affordance aspect.

Mapping:
We tackled the mapping principle by having titles on all the setup pages. All buttons are clearly labelled so that there is no confusion for the user regarding where they can go or what they can do on a page / in a scene.

Consistency:
We achieve consistency in our menu by having the standard title, back button and next button template for every page. We also achieve consistency across all of our scenes by having the exact same drop down menu in all of them for additional functions. We also consistently use the same colour scheme across our whole program.

Constraints:
An example of how we managed to satisfy the constraints principle in our design is by not allowing a user to move through walls or objects in our scenes. Another example is that we also don’t allow people to choose multiple scenes on the selection screen by only allowing one scene toggle to be chosen at a time.

Sketches


Initial Sketches

We presented our users with the following sketches based on alternative design styles for our software so we could obtain feedback and expand on the most popular sketches.​

​ For sketch 1.1 they felt there was too much happening with the image and block of text. They wanted a simpler approach to this screen.​

For sketch 1.2 they believed it would be very difficult to find the exact scene they wanted with no filtering option, but they did like the more minimalist design.​

For sketch 1.3 they didn't like the way the camera settings menu obscured the view of the scene they were dealing with.




Improved Sketches

We took the comments regarding minimalism heavily on board and hence decided to apply this style to all of our further sketches. For example, the welcome page was simplified by reducing unnecessary text and retaining only necessary buttons using icons instead of text where possible.​

The scene selection was improved by putting scenes into categories making specific ones easier to find.

​ On the camera settings page we made the preview a smaller window instead of a full screen embedded preview to give the user a complete and unobstructed view for easier calibration.

System Design



System Architecture Diagrams


Essentially, the way our software works is as follows:
The host presenter runs our Unity program and selects a scene. If they want to add more presenters to the scene they click that option in the menu and this opens a video streaming website. The host simply sends that link to the people he wants to add. The host gets OBS to capture the unity scene they have chosen and also the video streaming website. They crop and overlay the people in the video streaming web page using OBS and finally they choose OBS as their virtual camera in another video conferencing software (e.g. Microsoft Teams).


Our web app utilises P2P to share media between users.

Its structure consist of 2 servers:

-a relay and UI server which shares information about present peers, UI and the bodypix model to all peers
-a P2P brokering server which allows peers to share media streams with each other using the peer information provided by the relay server.
