Testing Overview

We utilised several methods of testing to instil confidence in our software. Due to the Unity nature of our application, most of our testing was manual combining human effort with heuristics to draw conclusions.

Our test types included:

  • Compatibility testing

  • Responsive design testing

  • Stress testing

  • Alpha testing (User Acceptance Testing)

  • Beta testing (User Acceptance Testing)

Compatibility testing

Our aims with compatibility testing was to realise what sorts of machines our software would work on. Our measures of performance was FPS for the Unity desktop application and noticeable lag for our automatic background removal.

In our testing we utilised 4 different machines (A to E) ranging from high end to very low end. The specifications of each machine can be found in here in our appendix.

In general:

Machine A: High end
Machine B: Mid/high end
Machine C: Mid/low end
Machine D: Low end
Machine E: Very low end

Our results showed that all machines (A to E) showed good performance achieving 30+ FPS in our Unity application.

For background removal our results were more varied. Machines (A to C) performance well however machine D showed poor background removal. Machine E failed completely with no background removal and a frozen website.

Reponsive design

Our responsive design testing aimed to determine whether our UI would scale appropriately to different viewing configurations.

Our testing method for Unity was to launch the program with different resolutions and test whether all functionality was still working.

Initially we found that, although accounted for, our UI resolution scaling in Unity did not function correctly. However upon review and further development we corrected this and have confidence that all resolutions above 1080p will function completely correctly.

Our testing method for our website was to adjust window size and confirm that the text and video grids will adjust appropriately to ensure they are still in view and functional. Our system passed this test.

Stress testing

Stress testing our web app was quite tricky due to the nature of the system requiring genuine users from different networks. Due to the peer to peer nature of the service, connection between users would continue to function in the event of a server outage.

Our testing method was to limit our connection 20 megabytes per second up and down. From here we would start a video session and invite peers and view when “noticeable lag” would appear.

Our testing confirmed 5 peers would work stably on a standard 20/20 connection.

Alpha testing

Considering out project was heavily UI orientated, alpha testing formed the bulk of our testing. Although we conducted alpha testing throughout our project, as we roughly tested each feature after implementation we decided to conduct a more formal alpha testing of key risk areas when near project completition.

Our testing method hence consisted of identifying key risk sand conducting neccesary simulations to find and patch potential bugs.

The key risks we tested and actions we took after investigation are below:

- Being able to escape the designated area of each scene

We found if speed is sufficinetly increased, penetration of walls is still possible. To minimise this we increased the radius of our object detection algorithm to increase the liklihood an object will be detected before penetration. Additionally some overlooked objects were missing mesh colliders which we added.

- Improper functioning of save/load

Our save/load fortunately functioned properly.

- Improper functioning of buttons/ui when subject to unnatural behaviour (spam)

We identified that in the scene selection screen, spamming the scene names could cause the image preview to stop functioning. We also found after accessing the drop down menu by pressing the cog once, it no longer highlights on cover. We are currently looking into these bugs however due to their rarity and low impact they are not a priortiy.

- Failure in establishing peer to peer connection on video site

We found that in the case that a peer to peer connection request is missed by a client, the connection is never established. We corrected this by adding a retry connection function. We also found that in some cases, even when a failed connection is reattempted, connection is never established. We suspect a bug in the peerjs library and will be making the neccesary github issue. To counter this issue we added a function to identify when it takes place, warn the user and automatically refresh the web page creating a new peerid for that user fixing the issue.

- Memory leaks

Fortunately, we were able to fix a significant issue when investigation this risk. We found that the relay server had been configured to continually send the bodypix model to a user, hence causing resources to continually build up and eventually crash the site for that user. We corrected this by restructuring the algorithm so that the model is only sent once.

Beta testing

Although minimal user acceptance testing by real users took place, we found the practice very valuable in the issues it corrected.

Testers:

Tester A - Secondary School Teacher
Tester B - University lecturer

Test cases and feedback:

  • From the main menu configure a scene to your liking

Tester A - It was nice and easy, I liked the design
Tester B - It was nice after we solved the issue but not being able to leave the load screen is a problem.

From this test we reevaluated our navigation buttons realising that we had missed various exit buttons which could be quite troublesome.

  • Stream a scene to Microsoft Teams using OBS

Tester A - It was really tricky and took quite long
Tester B - It was quite straight forward however I am already used to using OBS

From this test we concluded that we must improve the detail of our instructions especially when describing the OBS aspect.
